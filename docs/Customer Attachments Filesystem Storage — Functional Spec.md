# Customer Attachments Filesystem Storage — Functional Spec

## 0) Document Status
- Status: Agreed Functional Spec
- Scope Type: Backend + Desktop UI + Storage Policy
- Related Area: Customers / Attachments / Filesystem Storage / Customer Identity
- Execution Mode: Phase-by-phase implementation with closeout report + human test after each phase

---

## 1) Purpose
هذه الوثيقة تنظم **نقل سياسة مرفقات العملاء** من الاعتماد الأساسي على حفظ الملف داخل قاعدة البيانات إلى **سياسة تخزين ملفات فعلية داخل مجلدات على القرص**، مع ربطها بقاعدة البيانات كـ metadata فقط.

الهدف من هذا التحول:
- جعل المرفقات قابلة للفتح والوصول اليدوي بسهولة
- جعل مسار المرفقات ثابتًا وواضحًا داخل جذر المشروع / نفس مكان قاعدة البيانات
- منع تعارض أسماء العملاء أو المجلدات
- إنشاء **كود عميل عالمي غير قابل للتكرار** وصالح للاستخدام كاسم مجلد
- إتاحة زر واضح لفتح **مجلد مرفقات العميل** من شاشة العملاء
- الحفاظ على الربط الحقيقي بالمستخدم/العميل داخل السيرفر

---

## 2) Strategic Direction
هذه الوثيقة تغير **سياسة التخزين نفسها**، وليست مجرد تحسين UI.

### الوضع الحالي
المرفقات الحالية محفوظة بطريقة database/blob-oriented أو عبر مسار غير مناسب للوصول اليدوي.

### الوضع المطلوب
المرفقات تصبح محفوظة في **Filesystem folder structure** داخل جذر المشروع / نفس مكان قاعدة البيانات، مع الاحتفاظ في قاعدة البيانات فقط بالـ metadata والربط المنطقي.

---

## 3) Golden Rules
1. **Backend هو مصدر الحقيقة الوحيد**
   - لا يتم حفظ أي metadata أو path بشكل محلي فقط
2. **Filesystem storage policy must be real**
   - الملف نفسه يجب أن يوجد فعليًا في مجلد قابل للكتابة
3. **Customer code must be immutable**
   - كود العميل العالمي لا يتغير بعد إنشائه
   - ولا يعاد استخدامه حتى لو حذف العميل
4. **Safe path names only**
   - كود العميل المستخدم في المجلدات يجب أن يحتوي على:
     - أرقام
     - وشرطة `-` فقط
   - بدون `/` أو `\` أو أي رموز غير صالحة لمسارات الملفات
5. **No customer-name-based folders**
   - اسم المجلد لا يعتمد على اسم العميل
   - يعتمد فقط على الكود العالمي
6. **Readable and writable folder**
   - المجلد الفعلي يجب أن يكون قابلًا للكتابة
   - ويمكن فتحه من النظام عند الحاجة
7. **No fake filesystem UX**
   - لا يظهر زر “فتح مجلد المرفقات” إلا إذا كانت هناك طريقة حقيقية ومفهومة للوصول للمجلد
8. **Cleanup Audit mandatory**
   - كل مرحلة تنفيذية تحتوي على Cleanup Audit للملفات الملموسة فقط

---

## 4) Business Goal
الهدف التجاري من هذه العائلة:
- جعل مرفقات العميل الآجل/العميل التجاري قابلة للأرشفة الحقيقية
- تسهيل المراجعة القانونية/الإدارية
- تسهيل نسخ الملفات احتياطيًا
- ربط المرفقات بالعميل بشكل ثابت وواضح
- فتح المجلد مباشرة من شاشة العملاء عند الحاجة

---

## 5) Root Storage Policy

## 5.1 Root path
المجلد الجذري للمرفقات يجب أن يكون **داخل جذر المشروع / نفس مسار قاعدة البيانات**.

### القاعدة المعتمدة
إذا كان جذر البيانات/المشروع هو:
`<ProjectRoot>`  
فإن مسار المرفقات يكون مثلًا:

`<ProjectRoot>\CustomerAttachments\`

أو:
`<ProjectRoot>\Attachments\Customers\`

### القرار المعتمد في هذه الوثيقة
الاسم الموصى به هو:

`<ProjectRoot>\CustomerAttachments\`

لأنه:
- واضح
- مباشر
- خاص بالعملاء
- غير معقد

## 5.2 Path relation to database
المقصود بـ “نفس مكان قاعدة البيانات”:
- نفس الجذر/المجلد الأساسي الذي تُدار منه بيانات المشروع
- وليس مجلدًا عشوائيًا في جهاز المستخدم

---

## 6) Global Customer Code Policy

## 6.1 Main requirement
كل عميل يجب أن يمتلك **كودًا عالميًا غير قابل للتكرار** ويصلح ليكون اسم مجلد.

## 6.2 Character restrictions
الكود يجب أن يحتوي على:
- أرقام فقط
- وشرطة `-` فقط

### أمثلة مقبولة
- `2026-000001`
- `2026-000002`
- `2027-000315`

### أمثلة غير مقبولة
- `CUST/001`
- `Ahmed-001`
- `123\456`
- `A-001`

## 6.3 Immutability
- الكود ينشأ مرة واحدة فقط
- لا يتغير عند تعديل العميل
- لا يعاد استخدامه إذا حذف العميل
- يظل المرجع الثابت للمجلدات والمرفقات

## 6.4 Visibility
الكود يجب أن:
- يكون محفوظًا في السيرفر
- ويظهر عند الحاجة في تفاصيل العميل
- ويمكن استخدامه لاحقًا كمفتاح مرجعي للمراجعة والمرفقات

---

## 7) Folder Structure Policy

## 7.1 Per-customer folder
لكل عميل مجلد مستقل باسمه:
`<CustomerCode>`

### مثال
`<ProjectRoot>\CustomerAttachments\2026-000001\`

## 7.2 Optional subfolders
داخل مجلد العميل يمكن تنظيم المرفقات لاحقًا إلى:
- `NationalId`
- `Contracts`
- `Other`

لكن هذه الوثيقة لا تُلزم بالتقسيم الفرعي إذا كان implementation الأولي أبسط وأكثر استقرارًا بدونها.

### القاعدة
الأولوية الآن:
- ثبات التخزين
- صحة الربط
- سهولة الفتح
- عدم تضارب الأسماء

---

## 8) File Naming Policy

## 8.1 Stored file names
اسم الملف المخزن فعليًا يجب أن يكون آمنًا وصالحًا للـ filesystem.

### القاعدة
لا يعتمد الحفظ على الاسم الأصلي الخام وحده.

يجب حفظ:
- `OriginalFileName` للعرض
- و`StoredFileName` الآمن للاستخدام الفعلي على القرص

## 8.2 Allowed strategy
اسم الملف الفعلي المخزن يمكن أن يكون مثلًا:
- timestamp + short unique suffix + extension
- أو sequence داخلي آمن
- أو GUID-safe segment + extension

لكن:
- لا يجب أن يتعارض مع ملفات أخرى
- لا يجب أن يحتوي رموزًا غير صالحة
- لا يجب أن يضيع الامتداد

---

## 9) Database Metadata Policy

بدل حفظ الملف كـ blob كامل في DB، يتم الاحتفاظ ببيانات الربط فقط.

## 9.1 Required metadata per attachment
- AttachmentId
- CustomerId
- CustomerCode
- Category
- OriginalFileName
- StoredFileName
- RelativePath
- ContentType
- FileSize
- UploadedAt
- UploadedBy
- Notes (optional if supported)

## 9.2 No blob storage as source of truth
الملف الحقيقي يكون على القرص.
قاعدة البيانات تحتفظ بالـ metadata فقط.

## 9.3 Backward compatibility
إذا كانت هناك مرفقات قديمة مخزنة بطريقة مختلفة، يجب تحديد ذلك في خطة Migration ولا يُتجاهل.

---

## 10) Attachment Categories
الأنواع المستهدفة على الأقل:
- بطاقة رقم قومي
- تعاقد
- مرفق عام / أخرى

### ملاحظة
إذا كان النظام الحالي يدعم attachment type enum أو reason/category مختلف، يجب توثيق المابنج بوضوح.

## 11) Customer UI Requirements

## 11.1 Customer details
تفاصيل العميل يجب أن تعرض:
- الكود العالمي للعميل
- قائمة المرفقات
- اسم الملف
- نوع المرفق
- تاريخ الرفع
- المستخدم الذي رفعه
- أزرار:
  - فتح
  - تنزيل
  - فتح مجلد المرفقات (إن كان متاحًا من هذا السياق)

## 11.2 Customer list grid
في جدول العملاء يجب أن يوجد زر واضح بجانب السطر الخاص بالعميل:
- **فتح مجلد المرفقات**

### القاعدة
هذا الزر يفتح مجلد العميل نفسه، وليس ملفًا واحدًا.

### سلوك الزر
- إذا كان المجلد موجودًا:
  - يفتح مباشرة
- إذا لم يوجد:
  - رسالة واضحة
- إذا كان المسار غير متاح من البيئة الحالية:
  - رسالة صادقة ومفهومة

---

## 12) Open Folder Behavior

## 12.1 Primary action
زر **فتح مجلد المرفقات** يفتح:
`<ProjectRoot>\CustomerAttachments\<CustomerCode>\`

## 12.2 Safety
لا يتم توليد المسار من اسم العميل.
فقط من:
- root storage path
- customer code

## 12.3 Environment awareness
إذا كانت بيئة التشغيل لا تسمح بفتح المجلد مباشرة من جهاز العميل:
- لا يتم الادعاء بأن المجلد فتح
- بل تظهر رسالة واضحة تقول إن المسار غير متاح من البيئة الحالية

---

## 13) Migration Direction

## 13.1 Need for migration
إذا كان النظام الحالي يستخدم blob/database storage أو مسارًا أقدم، فهذه العائلة تحتاج خطة Migration.

## 13.2 Migration strategy options
### Option A — Big switch
- نقل كل المرفقات القديمة
- ثم اعتماد filesystem storage فقط

### Option B — Safe staged migration
- أي مرفق جديد يُحفظ بالنظام الجديد
- المرفقات القديمة يتم نقلها تدريجيًا
- النظام يظل قادرًا على فتح القديم والجديد لحين اكتمال النقل

### القرار الموصى به
**Safe staged migration**  
لأنها أقل خطورة وأكثر أمانًا على المشروع.

## 14) Phase Breakdown

# Phase CAFS-1 — Customer Global Code + Filesystem Storage Foundation
## Scope
- إضافة/تثبيت CustomerCode عالمي immutable
- Backend support لتخزين المرفقات كملفات على القرص
- حفظ metadata في قاعدة البيانات
- تعريف root path
- إنشاء مجلد العميل تلقائيًا
- حفظ المرفق داخل المجلد الصحيح
- الحفاظ على الربط بالعميل
- عدم استخدام اسم العميل كاسم مجلد

## Must Not Include
- فتح مجلد من الجدول
- تحسينات UI النهائية للمرفقات
- migration الكامل للقديم
- supplier storage

## Human Review After CAFS-1
راجع:
1. هل لكل عميل كود عالمي؟
2. هل الكود لا يحتوي إلا على أرقام وشرطة؟
3. هل المرفقات الجديدة تحفظ فعليًا داخل مجلد العميل؟
4. هل المجلد داخل جذر المشروع/جوار قاعدة البيانات؟
5. هل الملفات لا تتعارض أسماؤها؟

---

# Phase CAFS-2 — Customer Attachments Access + Open Folder Action
## Scope
- إظهار المرفقات بوضوح في تفاصيل العميل
- فتح/تنزيل الملفات
- زر **فتح مجلد المرفقات** في صف العميل داخل الجدول
- عرض الكود العالمي بشكل مناسب
- رسائل واضحة عند غياب المجلد أو تعذر الوصول

## Must Not Include
- migration الكامل
- supplier version
- whole-app attachment rollout

## Human Review After CAFS-2
راجع:
1. هل أرى المرفقات داخل تفاصيل العميل؟
2. هل يمكنني فتح/تنزيل المرفق؟
3. هل زر فتح مجلد المرفقات يعمل من جدول العملاء؟
4. هل يفتح المجلد الصحيح للعميل الصحيح؟
5. هل الكود العالمي ظاهر وسليم؟

---

# Phase CAFS-3 — Legacy Migration / Backfill (Optional / If Needed)
## Scope
- نقل المرفقات القديمة
- توحيد المسارات
- تنظيف المسار القديم
- توثيق ما تم نقله

## Must Not Include
- إعادة تصميم صفحات البرنامج كلها
- أي توسعة خارج العملاء

---

## 15) Human Acceptance for Full Closure
تعتبر هذه العائلة مغلقة عندما أستطيع بشريًا:
1. فتح عميل ورؤية كوده العالمي
2. التأكد أن الكود ثابت وغير قابل للتكرار
3. رفع مرفق جديد
4. العثور عليه داخل مجلد العميل الفعلي
5. فتح/تنزيل المرفق من تفاصيل العميل
6. فتح مجلد المرفقات من جدول العملاء
7. التأكد أن اسم المجلد صالح للكتابة ولا يحتوي رموزًا ممنوعة
8. عدم تعارض الملفات بين عملاء مختلفين
9. عدم ضياع الارتباط بعد التعديل أو إعادة الفتح

---

## 16) Non-Scope
هذه الأشياء خارج نطاق هذه الوثيقة:
- دفعات العملاء
- كشف الدفعات
- له / عليه
- دفعات مسبقة
- الموردين
- Print Config
- whole-app storage policy لكل الكيانات
- redesign شامل لكل البرنامج

---

## 17) Cleanup Policy
في كل مرحلة تنفيذية من هذه الوثيقة:
- يتم تنظيف الملفات الملموسة فقط
- حذف dead code الآمن فقط
- لا يتم whole-project cleanup
- كل closeout يجب أن يحتوي:
  - `## Cleanup Audit`
  - what was found
  - what was removed
  - why safe
  - what was intentionally left
  - build/test verification

---

## 18) Prompting Model
بعد اعتماد هذه الوثيقة:
- نبدأ بـ **CAFS-1**
- بعد نجاحها بشريًا ننتقل إلى **CAFS-2**
- لا يتم دمج المرحلتين في Prompt واحد

---

## 19) Final Rule
أي تعارض بين هذه الوثيقة وبين backend الحقيقة:
- يتم توثيقه بوضوح
- ويتبع التنفيذ backend truth
- وإذا احتاج الأمر migration staged، يتم تنفيذه تدريجيًا
- بدون اختراع حفظ وهمي أو زر يفتح مجلدًا غير موجود فعليًا
