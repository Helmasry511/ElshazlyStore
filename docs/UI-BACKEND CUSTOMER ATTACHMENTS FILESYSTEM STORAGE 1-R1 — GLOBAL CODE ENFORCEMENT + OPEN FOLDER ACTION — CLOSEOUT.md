## 1) Root Cause of the Visible-Code vs Internal-Code Mismatch

### What happened in CAFS-1
CAFS-1 introduced a new `CustomerCode` field (format: `YYYY-NNNNNN`, digits and `-` only, immutable, auto-generated) specifically for filesystem folder naming.  However, it **preserved the old `Code` field unchanged**:
- Old `Code` was mutable (editable on update)
- Old `Code` accepted ANY characters including `@@#!$$`
- Old `Code` was the ONLY code shown in the grid, details modal, and create/edit forms
- New `CustomerCode` was **never displayed anywhere in the UI**

### Why the user saw the problem
The user saw the `Code` field in the form, entered `@@#!$$`, and it was accepted. From the user's perspective, that WAS the customer code — the authoritative filesystem code `CustomerCode` was completely invisible.

### Root cause
CAFS-1 treated `CustomerCode` as a strictly internal/backend field and did not update the UI to make it the primary visible code. The old `Code` field retained full editability with zero character validation.

## 2) Final Authoritative Customer Code Policy

### Is there one customer code now or two?
**Two**, but with clear hierarchy:

| Field | Purpose | Format | Mutable | Visible |
|-------|---------|--------|---------|---------|
| `CustomerCode` | **Authoritative global code** — filesystem folder identity, attachment storage key | `YYYY-NNNNNN` (digits + `-` only) | ❌ Immutable after creation | ✅ **Primary visible code** in grid + details |
| `Code` | Legacy business code — used in accounting/payment party references | `CUST-NNNNNN` (auto-generated) | ❌ Immutable after creation (changed in this phase) | Shown read-only in edit form only |

### Which one is authoritative for storage identity?
**`CustomerCode`** (`YYYY-NNNNNN`) is the authoritative code for:
- Filesystem folder naming (`<RootPath>/<CustomerCode>/`)
- Attachment storage path identity
- Global customer identification

### What does the user now see?
- **Grid**: Shows `CustomerCode` (`YYYY-NNNNNN`) as the code column — was previously showing `Code`
- **Details modal**: Shows `CustomerCode` in the identity card — was previously showing `Code`
- **Create form**: Code field is **hidden** — both `Code` and `CustomerCode` are auto-generated by the server
- **Edit form**: Code field is shown **read-only** with hint "الكود يُولَّد تلقائيًا ولا يمكن تعديله"
- **Grid row**: New "📂 المرفقات" button opens the customer's attachments folder

### Explicit statement
**Invalid codes like `@@#!$$` are no longer acceptable as the authoritative global code.** The user cannot manually enter any customer code. Both `Code` and `CustomerCode` are server-generated with safe character sets. The `CustomerCode` used for storage identity only contains digits and hyphens.

## 3) Exact Validation/Enforcement Behavior

### Backend validation (CreateAsync)
If a `Code` value is provided via API (not from UI form, which now sends `null`):
- Each character is validated: only letters, digits, and `-` are allowed
- Invalid characters produce: `400 ValidationFailed "Customer code contains invalid character '{ch}'."`

### Backend validation (CustomerCode)
- Auto-generated server-side in `YYYY-NNNNNN` format
- `CustomerAttachmentStorageService.ValidateCustomerCode()` validates digits + `-` only before any filesystem operation
- Unique index in PostgreSQL: `IX_customers_CustomerCode`

### Desktop enforcement
- Create form: `Code` field is hidden entirely — `null` sent to server, server auto-generates
- Edit form: `Code` field is read-only (`IsReadOnly="True"`)
- No user input pathway exists to provide a custom code value

---

## 4) Exact Immutability Behavior

### CustomerCode (YYYY-NNNNNN)
- Generated once during `CreateAsync` — never changes
- Not present in `UpdateCustomerRequest` — cannot be set via update API
- Not modifiable from any UI pathway

### Code (CUST-NNNNNN)
- **Changed in this phase**: `UpdateAsync` now ignores `Code` even if provided in the request body
- Previously: code could be changed on update if the new value was unique
- Now: the `if (req.Code is not null)` block is a no-op with a comment explaining immutability
- Desktop sends `Code = null` on both create and update

## 5) Exact Open-Folder Action Behavior

### New API endpoint
`GET /api/v1/customers/{id}/attachments-folder`
- Returns `{ folderPath: string, exists: bool }`
- `folderPath`: Full filesystem path `<RootPath>/<CustomerCode>`
- `exists`: Whether the directory exists on disk
- Requires `CustomersRead` permission

### Desktop "Open Attachments Folder" button
- Location: Customer grid row, between "Details" and "Payments" buttons
- Label: "📂 المرفقات"
- Tooltip: "فتح مجلد مرفقات العميل"
- Behavior:
  1. Calls `GET /api/v1/customers/{id}/attachments-folder`
  2. If API fails: shows error notification "لا يمكن الوصول إلى مجلد المرفقات من هذه البيئة"
  3. If folder doesn't exist: shows info notification "لم يُنشأ مجلد مرفقات لهذا العميل بعد — سيُنشأ تلقائيًا عند رفع أول مرفق"
  4. If folder exists: opens it with `Process.Start(UseShellExecute = true)`
  5. If `Process.Start` throws: shows error notification

### Folder-path resolution rule
`<AttachmentStorage:RootPath>/<CustomerCode>/`

Example: `C:\...\bin\Debug\net8.0\CustomerAttachments\2026-000001\`
